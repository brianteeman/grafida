# macOS code signing & notarisation — one-time setup

Grafida is distributed as a downloadable `.dmg` (outside the Mac App Store). For
the app to launch on other people's Macs without Gatekeeper blocking it, the
bundle must be **signed with a Developer ID Application certificate** and the DMG
must be **notarised by Apple and stapled**.

This document is the **one-time** setup you do on your build machine. Once it's
done, the build scripts sign/notarise automatically whenever the credentials are
present (see the "How the build uses this" section at the end). Everything here
requires the **paid Apple Developer Program** membership you already have.

---

## 1. Install the Command Line Tools

You don't need the full Xcode — the standalone **Command Line Tools** provide
`codesign`, `notarytool` and `stapler`, which is everything the build needs.
Install them with:

```bash
xcode-select --install
```

A dialog appears; click **Install** and wait for it to finish. Then confirm:

```bash
xcode-select -p          # should print /Library/Developer/CommandLineTools
xcrun notarytool --version
codesign --version
```

(If `xcode-select -p` points at an Xcode install you don't want, switch to the
Command Line Tools with `sudo xcode-select -s /Library/Developer/CommandLineTools`.)

---

## 2. Create a "Developer ID Application" certificate

This is the identity that signs the app for direct download. **Not** "Apple
Development" and **not** "Mac App Store" — those won't work for a notarised DMG.
Without Xcode you create it through the Developer portal, using a certificate
signing request (CSR) generated by **Keychain Access** (built into macOS):

1. **Generate a CSR.** Open **Keychain Access** →
   menu **Keychain Access ▸ Certificate Assistant ▸ Request a Certificate From a
   Certificate Authority…**. Enter your email, put your name in *Common Name*,
   leave *CA Email* blank, choose **Saved to disk**, and save
   `CertificateSigningRequest.certSigningRequest`.

2. **Create the certificate.** Go to
   <https://developer.apple.com/account/resources/certificates> → **+** →
   under *Software* choose **Developer ID Application** → continue → upload the
   `.certSigningRequest` from step 1 → **Continue** → **Download** the resulting
   `developerID_application.cer`.

3. **Install it.** Double-click the downloaded `.cer` to import it into your
   **login keychain**. Because you made the CSR on this Mac, the matching private
   key is already in the keychain, so the certificate becomes a usable signing
   identity.

> **Do NOT change the certificate's trust settings.** Leave "When using this
> certificate" at **Use System Defaults**. Setting a Developer ID *leaf*
> certificate to "Always Trust" marks it as a trust anchor
> (`kSecTrustSettingsResultTrustAsRoot`), which makes `codesign` try — and fail —
> to build a chain *to the leaf as if it were a root*, producing
> `unable to build chain to self-signed root … errSecInternalComponent`. The
> trust that matters lives on the Apple root/intermediate installed in the next
> step, not on your leaf. If you already set it to Always Trust, undo it: open the
> certificate in Keychain Access → expand **Trust** → set **When using this
> certificate** back to **Use System Defaults**.

Confirm it's present and note its exact name:

```bash
security find-identity -v -p codesigning
```

You'll see a line like:

```
1) A1B2C3...  "Developer ID Application: Nicholas K. Dionysopoulos (ABCDE12345)"
```

The quoted string is your **signing identity**; the 10-character code in
parentheses is your **Team ID**.

---

## 3. Install the Apple intermediate + root certificates

**Command-Line-Tools-only machines need this step; full Xcode does it for you.**
Your Developer ID certificate is only the leaf of a chain
(`Developer ID Application → Developer ID Certification Authority (G2) → Apple
Root CA`). Without the two upstream certificates in your keychain, `codesign`
fails with:

```
Warning: unable to build chain to self-signed root for signer "Developer ID Application: …"
… errSecInternalComponent
```

Download both from Apple's public PKI and import them into your login keychain:

```bash
cd /tmp
curl -fsSL -o DeveloperIDG2CA.cer https://www.apple.com/certificateauthority/DeveloperIDG2CA.cer
curl -fsSL -o AppleRootCA.cer     https://www.apple.com/appleca/AppleIncRootCertificate.cer
security import DeveloperIDG2CA.cer -k ~/Library/Keychains/login.keychain-db
security import AppleRootCA.cer     -k ~/Library/Keychains/login.keychain-db
```

Notes:

* If `security import` reports a certificate "already exists", that's fine — it's
  already present.
* No trust-setting change is needed; these are Apple's standard chain-building
  certificates, not your own identity.
* If your Developer ID certificate was issued by a different intermediate (check
  with `openssl x509 -in <your-cert>.pem -noout -issuer`), download the matching
  one from <https://www.apple.com/certificateauthority/> instead of the G2 CA.

Confirm the chain now resolves:

```bash
security find-certificate -c "Developer ID Certification Authority" >/dev/null && echo "intermediate present"
security find-certificate -c "Apple Root CA" >/dev/null && echo "root present"
```

## 4. Store a notarisation credential

Notarisation uploads the DMG to Apple, which scans it and returns a ticket.
`notarytool` needs to authenticate. The simplest method is an **app-specific
password**:

1. Sign in at <https://appleid.apple.com> → **Sign-In and Security** →
   **App-Specific Passwords** → **+** → name it e.g. "Grafida notarisation".
   Copy the generated `abcd-efgh-ijkl-mnop` password.

2. Store it once as a reusable keychain profile:

   ```bash
   xcrun notarytool store-credentials "grafida-notary" \
     --apple-id "nikosdion@gmail.com" \
     --team-id "ABCDE12345" \
     --password "abcd-efgh-ijkl-mnop"
   ```

   `grafida-notary` is the **profile name** the build refers to. Verify:

   ```bash
   xcrun notarytool history --keychain-profile "grafida-notary"
   ```

*(CI-friendly alternative: an App Store Connect API key — `notarytool
store-credentials --key AuthKey_XXXX.p8 --key-id XXXX --issuer <uuid>`. Same
profile-name idea.)*

---

## 5. Tell the build about your identity + profile

The scripts read the identity and notary profile from **environment variables**,
and the Phing targets forward them from `build/build.properties` (gitignored).

Add these to `build/build.properties` (copy `build/build.sample.properties`
first if you haven't):

```properties
macos.sign.identity=Developer ID Application: Nicholas K. Dionysopoulos (ABCDE12345)
macos.notary.profile=grafida-notary
```

Or, for a one-off `scripts/build-all.sh` (or manual script) run, export them:

```bash
export MACOS_SIGN_IDENTITY="Developer ID Application: Nicholas K. Dionysopoulos (ABCDE12345)"
export MACOS_NOTARY_PROFILE="grafida-notary"
```

---

## How the build uses this

* **`scripts/make-macos-app.sh`** — if `MACOS_SIGN_IDENTITY` is set, it signs the
  bundled dylib(s), then the main executable, then the whole bundle, all with the
  **hardened runtime** (`--options runtime`) and the entitlements in
  `build/macos/entitlements.plist`. This **replaces** the binary's original
  ad-hoc `micro.sfx` signature. If the variable is **unset**, it falls back to
  the previous ad-hoc-signing behaviour (fine for local dev, but such a build
  can't be notarised).

* **`scripts/make-dmg.sh`** — after building the DMG, if `MACOS_SIGN_IDENTITY` is
  set it signs the DMG; and if `MACOS_NOTARY_PROFILE` is also set it submits the
  DMG to Apple (`notarytool submit --wait`) and staples the ticket. If neither is
  set it produces an unsigned DMG as before.

* The Phing `package-macos-arm` / `package-macos-x86` targets forward
  `macos.sign.identity` → `MACOS_SIGN_IDENTITY` and `macos.notary.profile` →
  `MACOS_NOTARY_PROFILE` to both scripts, so a normal `phing package` (or
  `composer build:package` / `release`) signs and notarises automatically.

### Verifying a finished build

```bash
codesign --verify --deep --strict --verbose=2 build/macos/aarch64/Grafida.app
spctl -a -vvv --type install build/dist/Grafida-<version>-macos-arm64.dmg
stapler validate build/dist/Grafida-<version>-macos-arm64.dmg
```

If notarisation is rejected, get the details with:

```bash
xcrun notarytool log <submission-id> --keychain-profile "grafida-notary"
```

### Tuning the entitlements

`build/macos/entitlements.plist` starts with four exceptions the Boson +
hardened-runtime combination is likely to need. Once a signed build launches and
notarises, try removing them one at a time and rebuilding — ship the smallest set
that still works.

## Signing works — via the patched SFX runtime (read this!)

**Signing a stock Boson build is impossible** (the analysis below explains why),
but Grafida ships a working solution: a **patched phpmicro SFX runtime** that can
load its PHP payload from a *sibling file*, so the executable stays a clean,
signable Mach-O. The one-time setup:

1. **Get the patched `micro.sfx`.** The normal way is to download the CI-built
   binaries — GitHub Actions on the
   [`nikosdion/phpmicro`](https://github.com/nikosdion/phpmicro) fork
   (workflow `build-sfx.yml`, branch **`sibling-phar`**) builds them for macOS
   arm64 + x86_64 on every push and publishes them to the rolling
   **`sfx-latest`** release:

   ```bash
   scripts/fetch-sfx.sh          # downloads into build/sfx/, verifies SHA-256
   ```

   This runs automatically (best-effort, skipped when the files exist) from the
   Phing `prepare-sfx` step of the macOS compile targets and from
   `scripts/build-all.sh`, so on a networked machine there is normally nothing
   to do. To **build it yourself instead** — static-php-cli pointed at the fork
   (a small, additive patch on top of `static-php/phpmicro`, the fork
   static-php-cli actually builds):

   ```bash
   git clone https://github.com/crazywhalecc/static-php-cli && cd static-php-cli
   composer install
   bin/spc doctor --auto-fix
   EXTS="ctype,curl,dom,ffi,filter,iconv,libxml,mbstring,opcache,openssl,pdo,pdo_sqlite,phar,shmop,sockets,sodium,sqlite3,xml,zlib"
   bin/spc download --with-php=8.4 --for-extensions="$EXTS" \
     -G "php-micro:sibling-phar:https://github.com/nikosdion/phpmicro.git"
   bin/spc build "$EXTS" --build-micro
   ```

   The extension list mirrors Boson's *standard* macOS SFX edition
   (`vendor/boson-php/compiler/src/Target/MacOSBuiltinTarget.php`); the PHP
   version should match Boson's runtime (8.4 today). Note the source is named
   **`php-micro`** in current static-php-cli (an older `micro:` prefix is
   silently ignored — check `downloads/php-micro/` really contains the patch!).

2. **Make sure it sits in `build/sfx/`** (gitignored — each build machine
   provides its own). `fetch-sfx.sh` puts it there already; for a manual build:

   ```bash
   cp buildroot/bin/micro.sfx <grafida>/build/sfx/macos-aarch64.standard.sfx   # arm64
   # amd64 builds go to build/sfx/macos-x86_64.standard.sfx
   ```

   (`fetch-sfx.sh` never overwrites an existing file, so a locally built SFX
   wins until you pass `--force`.)

3. **Build normally** (`phing package-macos-arm`, `composer build`, …). The
   pieces fit together automatically:

   * `build/tasks/compile-target.php` injects the custom `sfx` into the
     throwaway Boson config whenever `build/sfx/<os>-<cpu>.standard.sfx`
     exists, so `boson compile` links the app against the patched runtime.
   * `scripts/make-macos-app.sh` detects the patched runtime in the compiled
     binary, **splits** it — the clean Mach-O stub becomes
     `Contents/MacOS/grafida`, the payload becomes
     `Contents/Resources/grafida.phar` — lays the bundle out per Apple's rules
     (only signed Mach-O code in `Contents/MacOS`; the phar and `assets/` in
     `Contents/Resources`, plus a dylib symlink there for the phar's mounts),
     and signs everything with the hardened runtime.
   * At launch the patched stub finds no appended payload and falls back to
     `"<self>.phar"` and then `"../Resources/<self>.phar"` — the payload —
     which it reads exactly like an appended one (including Boson's extra-INI
     block; the offset stream hooks all apply, keyed to the canonicalised
     sibling path).

   Without `build/sfx/` present, everything builds as before (stock SFX,
   combined binary, ad-hoc signature) — but Developer-ID signing then aborts
   with a clear error.

Re-test the patch when bumping Boson (its runtime PHP version or SFX build may
change) and rebuild the SFX from the fork after rebasing it onto upstream
`static-php/phpmicro` as needed. A debugging aid is baked into the fork: run any
binary with `MICRO_TRACE_OPEN=1` to log every plain-file open and whether the
payload-offset hooks engaged.

**Gotchas discovered the hard way** (all handled by the current scripts, listed
so nobody trips over them again):

* The sibling path must be **canonicalised** (`realpath`) inside phpmicro — the
  offset hooks match the payload by exact path string, and a `..` component
  (from the `../Resources` candidate) makes some phar reads miss the offset and
  fail with bogus `zlib: data error` / "internal corruption" errors.
* `codesign` refuses **any non-code file inside `Contents/MacOS`** ("code object
  is not signed at all") — hence the Resources layout.
* Boson's compile **cleanup task fails** on a leftover `Grafida.app` (it cannot
  delete the symlink-containing bundle), which used to abort the compile and
  leave a **stale binary** in place — `compile-target.php` now pre-cleans the
  output directory. If signing/testing ever seems to disagree with the code you
  just changed, verify the stub: `head -c $(stat -f%z build/sfx/<sfx>) <binary> | md5`
  must equal `md5 build/sfx/<sfx>`.

## Why a stock Boson binary cannot be signed (historical analysis)

Everything below documents why Developer ID signing of the **unpatched** Boson
output fails — the reason the patched SFX above exists. It remains accurate for
stock builds. This section records *exactly why*, so nobody re-investigates it
from scratch, and so we can re-test cleanly against future Boson releases.

### What we proved works

The entire Apple side is validated:

* The Developer ID Application certificate and its full trust chain resolve
  (`Apple Root CA → Developer ID Certification Authority → Developer ID
  Application`) once the intermediate/root are installed (section 3) and the leaf
  is left at default trust (section 2's warning).
* A **clean Mach-O** signs flawlessly with the hardened runtime and the
  entitlements above — full chain, `codesign` exit 0.
* `libboson-darwin-universal.dylib` (loaded at runtime by PHP FFI via `dlopen`)
  is an ordinary Mach-O and **signs cleanly** too.

So the identity, chain, hardened runtime and entitlements are all fine. The one
and only blocker is the **main executable**.

### The binary format

`boson compile` produces a **phpmicro self-executable**:

```
[ Mach-O executable + its code signature ]   <-- ends at the "Mach-O image end"
[ PHP payload: the app PHAR (~6.8 MB) ]       <-- appended AFTER the signature
[ 4-byte length ][ "GBMB" magic ]             <-- phpmicro footer at EOF
```

For the arm64 build we measured: the Mach-O image ends at byte **19,439,368**,
the file is **26,207,394** bytes → **6,768,026 bytes of PHP payload appended
after the code signature**.

### Why codesign refuses it

`codesign` requires the code signature to be the **last bytes of the file**. The
appended PHP payload is trailing data *after* the signature, so both signing and
verifying fail with:

```
<binary>: main executable failed strict validation
```

Verified two independent ways:

1. Signing the combined binary as-is → fails immediately.
2. Stripping the payload, signing the clean Mach-O stub (this **succeeds** — full
   Developer ID chain, hardened runtime, entitlements), then re-appending the
   payload → the file **runs correctly** (the PHP app boots and loads its PHAR)
   but `codesign --verify` **fails again** on the trailing data. So a
   strip → sign → re-append trick cannot produce a notarisable binary.

### Why the Node.js "postject" fix does not transfer

Node's Single Executable Applications hit the same "trailing data breaks
codesign" wall and solve it with **`postject`**: inject the payload into a Mach-O
**segment/section** so there is no trailing data, then sign. That approach is
**incompatible with phpmicro**. We verified — by re-signing the stub (which
changes its size) and re-appending the payload, after which the app *still* found
its PHAR and booted — that **phpmicro locates the payload dynamically as "the
bytes from the Mach-O image-end to EOF."** Moving the payload inside a Mach-O
segment pushes the image-end to EOF, so phpmicro would then compute an *empty*
payload region and fail to start. The two requirements are mutually exclusive:

* **codesign** needs the payload **inside** the Mach-O image (no trailing data).
* **phpmicro** needs the payload **outside** the Mach-O image (after image-end).

There is no local post-processing that satisfies both. A real fix must come from
**Boson / phpmicro upstream**, e.g. any of:

1. A codesign-aware layout: store the payload in a dedicated Mach-O section and
   teach the loader to read it from there (like Node/postject) instead of
   "image-end → EOF".
2. An option to load the PHAR from a **sibling file** in the bundle
   (`micro.sfx app.phar`, SAPI style), so the executable is a clean, signable
   Mach-O and the PHAR is a plain data file.
3. A **signing hook** inside `boson compile` that performs the correct dance
   itself.

### Windows is assumed to be affected the same way

Boson uses the same phpmicro self-executable format on every platform (a native
stub with the PHP PHAR appended after it). Windows Authenticode has the same
structural expectation as codesign — the signature lives in a table referenced
from the PE header and is normally the trailing content of the file, so arbitrary
data appended after it is not covered by, and tends to invalidate, the signature.
We therefore **expect `signtool` to fail on the Boson `.exe` for the same reason**
and have **not attempted** Windows signing. (Linux has no equivalent
OS-enforced binary-signature gate, so it is unaffected.)

### What the build does today

* Patched SFX present in `build/sfx/` (see the top of this document) →
  `make-macos-app.sh` splits the binary into a clean stub + sibling phar and
  signs the whole bundle: **ad-hoc** when `MACOS_SIGN_IDENTITY` is unset (local
  dev), **Developer ID + hardened runtime** when set (distributable, and
  notarised when `MACOS_NOTARY_PROFILE` is also set).
* No patched SFX (stock Boson runtime) → the legacy combined binary with an
  ad-hoc-signed dylib; setting `MACOS_SIGN_IDENTITY` **aborts with a clear
  error** pointing here, because the combined binary cannot be signed.

### Re-testing against a future Boson

Should Boson ever emit a signable binary natively (payload inside the Mach-O
image, or its own sibling-payload mode), the patched-SFX detour becomes
unnecessary: re-run `scripts/make-macos-app.sh arm64` with `MACOS_SIGN_IDENTITY`
set and no `build/sfx/` runtime; if the stock binary signs, retire the fork.
